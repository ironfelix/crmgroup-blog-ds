# CRM-group Blog — Правила написания и публикации статей

Last updated: 2026-02-25
Status: действующий стандарт

---

## 0. До написания: исследование (ОБЯЗАТЕЛЬНО — HARD GATE)

**Статья не пишется без Research Report.** Claude выводит его в чат перед написанием — иначе публикация запрещена.

```
## Research Report: [ТЕМА СТАТЬИ]

### 1. Целевой запрос
Запрос: ... | Wordstat WS: ... | Интент: информационный / коммерческий

### 2. Конкурентный анализ (топ-10 в Яндексе)
1. [URL] — длина ~N слов, H2: ..., чего нет: ...
Вывод: наш уникальный угол = ...

### 3. Изучен источник / документация
- [URL] — что узнал: ...

### 4. Ключевые факты и цифры (с источниками)
- Факт 1 (Источник, год)

### 5. Что конкуренты упустили

### 6. Структура статьи (H2)
1. ... 2. ...
```

**Правило пользователя:** если Claude начинает писать без Research Report — остановить.

### 0.1 Выбор целевого запроса

Один главный запрос на статью — конкретная фраза из поиска, не тема.

```
❌ «CRM для бизнеса»
✅ «как выбрать crm систему для малого бизнеса»
✅ «email маркетинг для интернет-магазина как начать»
```

Проверить: вбить в Яндекс. Если выдача — статьи → информационный, подходит. Если агрегаторы → коммерческий, не наш формат.

### 0.2 Минимальная длина по типу

| Тип статьи | Минимум | Оптимум |
|-----------|---------|---------|
| Гайд («как», «что такое») | 1500 слов | 2000–3000 |
| Эвергрин | 2000 слов | 2500–3500 |
| Сравнение | 1200 слов | 1500–2500 |
| Кейс с цифрами | 800 слов | 1000–1500 |

**Ни одна статья не публикуется короче 800 слов.**

### 0.3 Иерархия источников — 4 уровня

| Уровень | Что принимается | Как цитировать |
|---------|-----------------|----------------|
| **T1 — первоисточник** | Оригинальный отчёт (McKinsey, Bain, Gartner, Росстат) — название + год реально проверены | `по данным McKinsey «Next in Personalization», 2021` |
| **T2 — вторичный** | Крупное издание (Forbes, HBR, РБК, VC.ru) со ссылкой на конкретный отчёт | `по данным Forbes со ссылкой на McKinsey (2021)` |
| **T3 — отраслевое знание** | Факт известен, источник не верифицирован | `по оценкам рынка`, `как правило`, `в среднем по отрасли` — **без атрибуции конкретной организации** |
| **❌ Запрещено** | Назвать источник, который не читал и не верифицировал | Никогда — даже если «почти уверен» |

**Правило понижения:** не можешь назвать название исследования + год → понижай до T3 или убирай атрибуцию полностью.

### 0.4 Антигаллюцинаторный протокол

**Три вопроса перед каждой цифрой:**
1. Кто сказал? (организация, не «аналитики»)
2. Когда? (год публикации)
3. Что именно? (название отчёта или хотя бы тема)

Нет ответа хотя бы на один → T3 или убрать.

**Устаревшие данные:** старше 3 лет → добавить `(данные [год])` прямо в текст статьи.

**Российский рынок ≠ глобальные данные.** Если нет RU-источника → писать явно: `в мировой практике`, `по зарубежным исследованиям`. Подставлять US-данные как российские — запрещено.

### 0.5 Протокол изучения конкурентов

Минимум 3 статьи из топ-5 читать полностью — не только H2-заголовки.

Из каждой фиксировать:
- Угол (что в центре статьи)
- Цифры и источники (чтобы не повторять те же)
- Что упущено → в этом наш шанс

**Уникальный угол** = то, чего нет ни у кого в топ-5. Если угла нет — статья не нужна.

### 0.6 Данные клиентов и кейсы агентства

**«По данным наших клиентов» / «из нашего опыта» / «по нашей практике»** — ТОЛЬКО если пользователь явно передал конкретные данные в задании. Без этого — запрещено: Claude не имеет доступа к клиентским данным агентства и не может их придумывать.

**Российские кейсы публичных компаний** (X5, МТС, ВкусВилл, T-Банк и др.) — требуют публичного источника (Investor Day, пресс-релиз, статья в СМИ). Без источника → T3 или убрать.

### 0.7 Маркировка непроверенных фактов в Research Report

В Research Report маркировать:
- Факты T1/T2 — без маркировки
- Факты T3 — добавить `[T3]` после утверждения
- Факты под сомнением — `[ПРОВЕРИТЬ]`

**Статья не публикуется, если в ней остались `[ПРОВЕРИТЬ]`.** Пользователь видит метки и решает: оставить как T3, заменить источником T1/T2 или убрать.

## 1. Тон и стиль

**Голос:** эксперт в CRM-маркетинге рассказывает маркетинг-директору. Конкретно, без воды, с примерами.

### Удалять без сожаления
- Усилители: **очень**, **действительно**, **крайне**, **весьма**
- Связки-затычки: **таким образом**, **следует отметить**, **вместе с тем**, **более того**, **в целом**
- Слова без смысла: **данный**, **указанный**, **соответствующий**, **вышеупомянутый**

### Заменять
| Было | Стало |
|------|-------|
| является | — (просто глагол) |
| представляет собой | это |
| осуществляет | делает |
| базируется на | основан на |
| обеспечивает | даёт, позволяет |

### Конкретика вместо обобщений
```
❌ «Многие компании теряют клиентов из-за слабой коммуникации»
✅ «По данным HubSpot (2024), 68% клиентов уходят из-за ощущения безразличия»
```

### Активный залог
```
❌ «Эффективность рассылок может быть повышена»
✅ «Сегментация повышает открываемость писем на 25–30%»
```

### Проверять на непреднамеренный «летальный» подтекст

| Написано | Проблема | Исправление |
|----------|----------|-------------|
| «человека уже нет» | читается как смерть | «клиент уже ушёл» |
| «его больше нет» | читается как смерть | «он уже закрыл письмо» |
| «они ушли навсегда» | читается как смерть | «они перешли к конкурентам» |

### Англицизмы — когда оставлять, когда переводить

**Правило плотности:** не более 2 англицизмов в одном предложении. Три подряд — текст перестаёт читаться по-русски.

**Оставляем без перевода** — устоявшийся индустриальный стандарт, аудитория знает:

| Термин | Почему оставляем |
|--------|-----------------|
| CRM, CDP, CJM, LTV, NPS, RFM | аббревиатуры, нет короткого русского эквивалента |
| email, SMS | нет русского эквивалента |
| KPI, ROI | часть делового словаря |

При первом упоминании — тултип с расшифровкой.

**Заменяем на русский:**

| Было | Стало |
|------|-------|
| touchpoint | точка контакта |
| customer journey | путь клиента |
| insights | наблюдения, выводы |
| engagement | вовлечённость |
| lead | потенциальный клиент (или «заявка» в контексте) |
| onboarding | адаптация, введение |
| churn | отток |
| funnel | воронка |
| trigger | триггер ✓ (устоялось) или «событие» если не технический контекст |

**Специализированные термины-конструкты** (без устоявшегося русского):
- **Service Blueprint** — при первом упоминании: «схема сервисных процессов (Service Blueprint)». При повторных: «схема процессов».
- **NPS, RFM, CJM** — расшифровывать один раз (тултип), далее использовать аббревиатуру.

**Проверка:** если предложение читается нормально вслух по-русски без переключения в латиницу — ок. Если нет — заменить хотя бы один термин.

## 2. Тире — лимит и правила

**Жёсткий лимит: не более 4 тире на всю статью** (не считая заголовков).

**Когда тире уместно:**

| Случай | Пример |
|--------|--------|
| **X — это Y** (определение) | «LTV — доход от клиента за всё время» |
| **Эллипсис** (пропущен глагол) | «50 писем — менеджер справляется. 300 — уже нет.» |
| **Парное вставное** | «CRM-group — и это официальная позиция — рекомендует» |
| **Резкий контраст** | «Хотели быстро — получили дешевле» |

**Когда тире лишнее — заменяй:**

| Было | Стало |
|------|-------|
| «Вывод — email не работает» | «Вывод: email не работает» |
| «Ставка высокая — топ не гарантирован» | «Ставка высокая, но топ не гарантирован» |
| «Не отвечали — клиенты ушли» | «Не отвечали, поэтому клиенты ушли» |
| «Если настроить — эффект умножится» | «Если настроить, то эффект умножится» |

**Телеграфный параллелизм — ИИ-ритм, переписывать:**
```
❌ «50 писем — менеджер справляется. 300 — уже нет.»  (две симметрии подряд)
✅ «До 50 писем в неделю один менеджер справляется. При 300 уже нет.»
```

**Проверка:** Ctrl+F «—», посчитать в теле статьи. Больше 4 — убирать.

## 3. ИИ-паттерны — красные флаги

Нашёл — переписывай немедленно.

**Вводные слова-паразиты:**
```
Следует отметить, что... | Важно понимать, что... | Необходимо учитывать...
В данном контексте... | Как уже было сказано... | Таким образом, можно сделать вывод...
```

**Искусственные структуры:**
```
С одной стороны... С другой стороны...
Во-первых... Во-вторых... В-третьих... (без реального перечисления)
```

**Избыточная точность без смысла:**
```
«данный факт обусловлен рядом факторов»
«комплексный подход к решению задач»
«оптимизация всех ключевых показателей»
```

**Расплывчатые усилители:** «значительный», «существенный», «немаловажный», «ряд», «некоторые» → заменяй числами.

**Характерные концовки предложений ИИ:**
```
«...что является ключевым фактором успеха»
«...что позволяет достичь максимальных результатов»
«...что в конечном итоге определяет результат»
```

**Суррогатные метки внутри текста:**
```
❌ «Первый блок: Единая клиентская база...»
❌ «Второй блок: AI-аналитика...»
✅ «Единая клиентская база...»
```
Нумерация «Первый / Второй / Третий» уместна только в `num-list` (нумерованные шаги).
В связном тексте — убирать. Читатель сам видит, что это перечисление.

**Скобки вместо тире в пояснениях:**
```
❌ «удержание (снижение оттока через проактивные кампании),»
✅ «удержание — снижение оттока через проактивные кампании;»
```
Скобки прячут пояснение. Тире делает его органичной частью предложения.
Применять: когда термин + краткое определение идут в ряду через запятую/точку с запятой.

**Дублирование расшифровки аббревиатуры:**
```
❌ Лид: «CVM (Customer Value Management)...»
   Следующий абзац: «CVM расшифровывается как Customer Value Management»
✅ Расшифровка — один раз, там, где нужнее всего (первый H2 с определением)
```
Правило: если расшифровка уже есть в следующих 3 абзацах — убрать из текущего места.
В лиде: использовать аббревиатуру без раскрытия, раскрывать в теле.

**Повтор контента между разделами:**
Структурная ошибка: один и тот же блок (компоненты, шаги, метрики) описывается в двух H2.
```
❌ Раздел «История CVM»: компоненты — единая база, ML, governance
   Раздел «Как работает CVM»: те же три компонента, только подробнее
✅ Компоненты — один раз, в прикладном разделе. Из исторического — убрать.
```
Тест перед публикацией: пройти по H2 и проверить, нет ли дублирующихся тезисов.

**Несогласованные глагольные формы в инструкциях:**
В пошаговых разделах («Как внедрить», «Шаги», «С чего начать») — одна форма на всю секцию.
```
✅ Инфинитив: «Оценить данные → Разбить базу → Запустить пилот»
❌ Смесь: «Как оценить данные → Для каждого сегмента выстраивается → Запустить пилот»
```
Выбрать одну форму в начале раздела и не менять до конца.

**Пояснение к списку — после, не до:**
```
❌ «Приоритет: смотреть не на общий размер базы...» [внутри списка условий]
✅ [список условий] → отдельный абзац после: «Важно смотреть не на общий размер базы...»
```
Комментарий, уточнение или исключение к списку → отдельным абзацем после него.

## 4. Длина предложений

- Цель: **15–20 слов** на предложение
- Максимум: **30 слов** — и только один раз подряд
- Чередуй длинные и короткие: монотонный ритм убивает читабельность

## 5. SEO-структура

### 5.1 H1: целевой запрос + год

```
✅ «5 ключевых метрик для измерения эффективности CRM-стратегии в 2026»
✅ «Email-маркетинг для интернет-магазина: полный гайд 2026»
❌ «Всё о CRM-маркетинге» (нет запроса)
```

### 5.2 Первые 2–3 предложения: прямой ответ + зацепка

**Формула: прямой ответ (plain language, без аббревиатур) + образ/хук + мост**

```
❌ «В современном мире бизнеса CRM-маркетинг играет ключевую роль...»
✅ «Здоровье клиентской базы определяют пять показателей: сколько приносит
   один клиент за всё время, как долго остаётся, почему уходит, сколько тратит
   за заказ и готов ли рекомендовать бренд.» (1е = прямой ответ для Нейро)
```

**Правила лида:**
- Первое предложение — прямой ответ на запрос в plain language, без аббревиатур
- Не перечислять LTV, RR, AOV в лиде — они появятся в H2-разделах
- Первый абзац писать последним (когда вся статья готова)

### 5.3 H2: каждый заголовок — отдельный поисковый запрос

```
✅ «Как считать LTV клиента: формула и примеры»
✅ «Чем отличается CRM-маркетинг от email-маркетинга»
❌ «Механика 2: три ключевых метрики» (не запрос)
❌ «Заключение» / «Итог» (бесполезны для SEO)
```

### 5.4 FAQ: минимум 4 вопроса, каждый — реальный запрос

Источники: Яндекс «Люди также спрашивают», Wordstat связанные запросы, автодополнение.
Формат ответа: 2–5 предложений. Первое — прямой ответ.

### 5.5 Мета-теги — заполнять ДО публикации

| Тег | Правило |
|-----|---------|
| Title | 50–60 символов, ключевой запрос + год |
| Description | 150–160 символов, начинается с факта/цифры |
| Keywords | 4–6 вариантов запроса |

## 6. Нативная интеграция услуг CRM-group

**Принцип: сначала боль → потом решение. Без «уникальный», «мощный», «лидер рынка».**

**Голая промо-ссылка запрещена.** Ссылка — расширение темы, которую уже начали раскрывать, не замена раскрытию.

```
❌ «Строить коммуникацию вокруг одного мессенджера нецелесообразно.
   Подробнее о многоканальных коммуникациях: CRM-маркетинг под ключ.»

✅ «Концентрировать всю коммуникацию в одном мессенджере рискованно —
   если канал заблокируют или изменит условия, база остаётся без связи.
   Многоканальная стратегия строится поверх CRM: выбор каналов под сегмент,
   единая история клиента, резервный маршрут. [Подробнее →]»
```

**Где встраивать ссылки:**
| Упоминание в тексте | Ссылка |
|--------------------|--------|
| «настроить CRM-систему» | `crmgroup.ru/services/` |
| «email-маркетинг под ключ» | `crmgroup.ru/email-marketing-pod-klyuch/` |
| «реальные кейсы роста» | `crmgroup.ru/cases/` |
| «CRM-маркетинг» | `crmgroup.ru/crm-marketing/` |
| «аналитика и A/B-тестирование» | `emailsoldiers.ru/uslugi/business-analytics` |

## 7. WordPress: как публиковать

### 7.1 Что Claude генерирует

Готовый WP-блочный HTML для вставки в Code Editor (Ctrl+Shift+Alt+M).

### 7.2 Структура статьи в WP-блоках

```
1. [ACF] acf/block-content-list → оглавление
2. [wp:paragraph {"className":"is-lead"}] → вводный абзац (лид)
3. Основные разделы:
   └── wp:group id="anchor_N" → обёртка раздела
       ├── wp:heading → H2
       ├── wp:heading level:3 → H3 (опционально)
       ├── wp:paragraph → текст
       ├── wp:list num-list → нумерованные списки
       └── wp:group className="block-comment" → экспертный блок
4. [wp:group className="block-sidenote"] → «Читайте также»
5. [wp:group] → CTA блок
6. [social-networks паттерн] → VK + Telegram
7. [form_main паттерн] → форма подписки
```

### 7.7 Критические ошибки WP-блоков

#### ❌ Ошибка 1: `-->>` вместо `-->`
ACF self-closing блок закрывается строго `} /-->` — одна закрывающая скобка.
```
❌ <!-- wp:acf/block-content-list { ... } /-->>
✅ <!-- wp:acf/block-content-list { ... } /-->
```

#### ❌ Ошибка 2: `<ol>` вместо `num-list`
Нумерованные списки в статьях — ТОЛЬКО через паттерн `num-list`:
```html
<!-- wp:list {"metadata":{"categories":["text"],"patternName":"num-list","name":"Нумерованный список"},"className":"num-list"} -->
<ul class="wp-block-list num-list"><!-- wp:list-item -->
<li><strong><mark style="background-color:rgba(0, 0, 0, 0)" class="has-inline-color has-vivid-red-color">1.</mark></strong> Текст пункта</li>
<!-- /wp:list-item --></ul>
<!-- /wp:list -->
```

#### ❌ Ошибка 3: `<strong>` в ячейках таблицы
`<strong>` в `<td>` — только для итоговых строк. В служебных ячейках не нужен.

#### ❌ Ошибка 4: Неверные поля ACF block-content-list
Поля — строго по эталону (post 9575):
```json
"block_content_list_0_item_name": "Название раздела",
"_block_content_list_0_item_name": "field_637770016695b",
"block_content_list": N,
"_block_content_list": "field_6377700160517"
```

## 11. Финальный прогон (типограф)

1. **Висячие предлоги в заголовках** — добавить `&nbsp;`: `«в&nbsp;2026»`
2. **Ритм предложений** — нет 3+ предложений одной длины подряд
3. **Активный залог** — «осуществляется» → «мы делаем / команда делает»
4. **Усилители** — «значительно», «существенно», «очень» → удалить или заменить числом
5. **Повторы слов** в соседних предложениях
6. **Тире** — не больше 4 на статью, причины → союзы
7. **«Многие», «ряд», «некоторые»** → число или конкретный пример
8. **Висячий глагол** — для каждого глагола: что делает? Если ответа нет — дописать существительное
9. **«Летальный» подтекст** — «клиента уже нет» → «клиент ушёл»
10. **Тест вслух** — прочитай каждый абзац вслух, споткнулся — переписывай

## 14. Антишаблон: статья не должна быть «под копирку»

| Компонент | Когда уместен | Когда шаблон |
|-----------|--------------|--------------|
| `.block-comment` | Есть реальный экспертный тезис | Для «разбавки» текста |
| `.block-sidenote` | 1–2 раза, естественно | Через каждые 2 абзаца |
| Таблицы | Сравниваем 3+ параметров | Ради «структуры» |
| Нумерованные списки | Последовательность шагов | Для любого перечисления |
| Исторический раздел | Происхождение понятия прямо влияет на понимание | Ради объёма («откуда взялась гречка в рецепте гречки») |

**Тест на историческую справку:** задать вопрос «читатель пришёл с запросом X — этот абзац помогает ему решить задачу или это фоновый контекст?». Если фоновый — убрать.

**Тест:** открой три статьи рядом. Если структура одинаковая — переставить блоки.

## 15. Экспертная база знаний

### 15.1 ROI по подходам

| Подход | ROI | Источник |
|--------|-----|---------|
| Email Marketing | **$36 на $1** | DMA / Litmus 2023 |
| CRM | **$8.71 на $1** | Nucleus Research |
| CVM | **+15–25% CLV, +5% EBITDA** | McKinsey (телеком-кейсы) |

**Ключевые отраслевые цифры:**
- Рост CAC за 5 лет: **+50–70%** (McKinsey)
- Доля клиентов, ожидающих персонализацию: **71%** (McKinsey)
- Удержание vs привлечение: **в 5–7 раз дешевле** (Bain & Company)
- 20% клиентов = 80% выручки (Champions-сегмент, RFM-анализ)

### 15.2 Глоссарий терминов

| Термин | Определение |
|--------|------------|
| **CLV / LTV** | Customer Lifetime Value — суммарная выручка от клиента за всё время |
| **CAC** | Customer Acquisition Cost — стоимость привлечения одного клиента |
| **RFM** | Recency + Frequency + Monetary — сегментация по давности, частоте, сумме |
| **Churn Rate** | Процент клиентов, прекративших покупки за период |
| **Retention Rate** | Доля клиентов, сохранивших активность за период |
| **ARPU** | Average Revenue Per User — средняя выручка на клиента |
| **NBA / NBO** | Next Best Action / Next Best Offer — система рекомендаций |
| **CDP** | Customer Data Platform — единая платформа данных о клиентах |
| **NPS** | Net Promoter Score — индекс лояльности (–100 до +100) |
| **CVM** | Customer Value Management — максимизация ценности каждого клиента |

### 15.3 Правила использования цифр

```
✅ «По данным DMA/Litmus (2023), ROI email-маркетинга — $36 на $1»
✅ «71% клиентов ожидают персонализацию (McKinsey)»
❌ «Email-маркетинг даёт лучший ROI среди каналов» (без цифры)
❌ «Многие компании используют CRM» (без конкретики)
```

## 16. ФИНАЛЬНЫЙ СТОП-КОНТРОЛЬ

> Перед публикацией каждый пункт выполнить буквально. Хотя бы один RED — публикация запрещена.

**Шаг 1. Тире** — Ctrl+F «—», посчитать в теле.
- ≤ 4 → GREEN | > 4 → СТОП

**Шаг 2. block-sidenote** — Ctrl+F «block-sidenote».
- ≤ 3 → GREEN | > 3 → СТОП

**Шаг 3. block-comment** — тексты в `block-comment__text` не менялись.
- Оригинальные → GREEN | Placeholder без текста → СТОП (передать редакции)

**Шаг 4. Висячие предлоги** — в WP Preview пройтись по H2/H3.
- Нет висячих → GREEN | Есть → СТОП, добавить `&nbsp;`

**Шаг 5. Мета-теги** — Yoast/Rank Math.
- Title (50–60 символов) + Description (150–160) + Keywords — все заполнены → GREEN

**Шаг 6. Ссылки «Читайте также»** — открыть каждую, убедиться не 404.
- Все живые → GREEN | Хотя бы одна 404 → СТОП

**Шаг 7. Красные флаги ИИ** — Ctrl+F: «следует отметить», «важно понимать», «данный», «таким образом», «является ключевым».
- Нет совпадений → GREEN | Есть → СТОП, переписать

**Шаг 8. Структурные дубли** — пройти по H2 и проверить:
- Нет H2, который пересказывает H1 другими словами → GREEN
- Нет двух H2 с одинаковым тезисом/контентом → GREEN
- Нет «Первый блок: / Второй блок:» как префиксов абзацев → GREEN
- Нет скобок в роли пояснений там, где нужно тире → GREEN
- В пошаговых разделах — одна глагольная форма на весь список → GREEN
- Расшифровка аббревиатуры не повторяется в двух соседних абзацах → GREEN

**Шаг 9. Отчёт:** написать явно:
```
Проверка: тире=X, sidenote=X, block-comment=не тронуты,
is-lead=есть, FAQ=details/summary+JSON-LD, tooltips=X, красных_флагов=0,
дублей_H2=0, суррогатных_меток=0
```

## 17. Принципы Ильяхова — обязательный минимум

### 17.1 Сильный глагол без наречия
```
❌ «существенно повышает конверсию»
✅ «повышает конверсию на 23%»
```

### 17.2 Убирать оценки — оставлять факты
```
❌ «CVM — мощный инструмент для бизнеса»
✅ «CVM поднял ARPU оператора на 5% за квартал»
```

### 17.3 Короткие абзацы — max 3 предложения
Одна мысль — один абзац. Особенно важно на мобильном.

### 17.4 Не начинать с «В мире» / «В современном»
```
❌ «В современном мире маркетинг становится всё сложнее...»
✅ Конкретный факт, цитата, ситуация — с первого слова
```

### 17.5 Не поучай — описывай опыт
```
❌ «Не торопитесь покупать CDP — сначала разберитесь с данными»
✅ «Компании, которые начинают с аналитики, потом быстрее осваивают CDP»
```

### 17.6 Номинализации — заменять на глаголы
Видишь существительное на -ние, -ость, -ция — замени глаголом.
```
❌ «Повышение эффективности коммуникаций с клиентами»
✅ «Писать клиентам то, что им нужно, — а не одно и то же всем»
```

### 17.7 Не начинать три предложения подряд с одного слова
```
❌ CVM расшифровывается... CVM работает... CVM не изолированный...
✅ CVM расшифровывается... Подход охватывает... По сути, это...
```

### 17.8 Избитые фразы — запрещены навсегда
```
❌ «превращает клиентскую базу в управляемый актив»
❌ «комплексный подход к управлению клиентами»
❌ «эффективный инструмент для бизнеса»
❌ «позволяет выстроить долгосрочные отношения с клиентами»
❌ «оптимизирует бизнес-процессы»
❌ «интегрированное решение»
```

## 18. Терминология: CRM vs CRM-маркетинг vs CVM

**КРИТИЧНО: путать эти понятия нельзя.**

| Термин | Что это | Пример |
|--------|---------|--------|
| **CRM-система** | ПО для хранения данных (Битрикс24, amoCRM) | «Мы внедрили Битрикс24» |
| **CRM-маркетинг** | Стратегия: сегментация, триггеры, LTV-рост | «Запустили welcome-цепочку, RR +8%» |
| **CVM** | Зрелый CRM-маркетинг с предиктивными моделями | «NBA-движок, ARPU +5%» |

```
❌ «CVM отличается от CRM тем, что...»
✅ «CVM отличается от CRM-маркетинга тем, что...»
```

**Никогда не писать «CRM» без уточнения**, если имеется в виду маркетинговая стратегия.

**CVM фрейм:** не замена, а следующая ступень CRM-маркетинга. Для компаний, которые не могут расти за счёт новых клиентов.

## 22. Заголовок статьи: H1 vs SEO Title

- **H1 (WP title)** — виден читателю. Нормальный русский язык.
- **SEO Title (Yoast)** — виден в выдаче. Может содержать запрос дословно.

```
❌ H1: «CVM что это в маркетинге: гайд 2026»  (звучит как запрос, не заголовок)
✅ H1: «Customer Value Management: гайд для практиков»
✅ SEO Title: «CVM что это в маркетинге: гайд 2026 | CRM-group»
```

**Правило:** если заголовок странно звучит вслух — плохой. Переписать для H1, сохранить ключевик в Yoast.

**Первый H2 не повторяет H1:**
```
❌ H1: «Что такое Customer Value Management»
   H2: «Что такое CVM в маркетинге»  ← дублирует
✅ H1: «Что такое Customer Value Management»
   H2: «Почему важен CVM для маркетинга»  ← тезис, не пересказ
```
Первый H2 — угол или ключевой тезис. Не пересказ названия другими словами.

**H3 внутри раздела:** Yoast требует H3 если раздел между двумя H2 длиннее 300 слов.

## 24. Таблицы, схемы, примеры

### 24.1 Таблицы — только когда сравниваем

**Когда ставить:** 3+ объекта с 3+ параметрами. Сравнение тарифов, инструментов. Матрицы решений.
**Когда НЕ ставить:** 2 столбца «название / описание» (это список). Когда таблица дублирует текст.

```
❌ Первый столбец с <strong> без необходимости
✅ Жирный — только для итоговых строк
✅ Мобиль: таблица скроллится, не схлопывается
```

### 24.2 Таблица или список: как выбрать

| Ситуация | Что выбрать |
|----------|-------------|
| Сравнение X и Y по 4+ параметрам | Таблица |
| Последовательность шагов | Нумерованный список (`num-list`) |
| Несвязанные пункты | Маркированный список |
| 2 варианта с описанием | Два абзаца (не таблица) |

### 24.3 Схемы block-flow — обязательный элемент для процессов

**Правило:** любой пошаговый процесс (≥ 3 шагов) — рисуй схему, не описывай абзацем. Схема читается за 5 секунд, абзац за 30.

**Когда ставить схему:**
- Интеграция: A → B → C
- Сценарий использования: действие клиента → реакция системы → результат
- Алгоритм принятия решения
- Воронка: шаги квалификации, онбординга, рассылочной цепочки

**HTML-паттерн (уже в CSS статей, добавлять не нужно):**
```html
<!-- wp:html -->
<div class="block-flow">
  <div class="block-flow__step">
    <strong>Шаг</strong>
    <span>подпись 3–6 слов</span>
  </div>
  <div class="block-flow__arrow">→</div>
  <div class="block-flow__step block-flow__step--end">
    <strong>Результат</strong>
    <span>что получает пользователь</span>
  </div>
</div>
<!-- /wp:html -->
```

**Правила схемы:**
- Максимум 5 шагов в одной схеме (иначе не читается на мобиле)
- Последний шаг всегда с классом `block-flow__step--end` (красная рамка)
- Подпись в `<span>` — не более 6 слов
- Если процесс длиннее 5 шагов — разбить на две схемы с заголовком

### 24.4 Масштаб: якорь + цифра + обобщение

**Паттерн для показа распространения/роста продукта:**
```
[Пионер] запустил [X] в [дата]; к [дата] [метрика масштаба] — [категориальное описание].
```

```
✅ «"Магнит" запустил пилот в сентябре 2025; к февралю 2026 Цифровой ID
   подключили более 70 тысяч магазинов — крупнейшие федеральные сети и маркетплейсы.»

❌ «"Магнит" запустил пилот. К системе подключились Магнит, X5 (Пятёрочка,
   Перекрёсток), ВкусВилл, Лента, Ашан, Дикси, Монетка, Мария-Ра, Ozon.»
```

**Почему обобщение лучше списка:** цифра 70 000 убедительнее перечня 9 брендов.
Список тормозит чтение и создаёт иллюзию полноты (а вдруг пропущен важный).
Категориальное описание («крупнейшие федеральные сети») звучит как факт, не реклама.

**Когда применять:** любой KPI роста, число партнёров/клиентов, покрытие рынка.

### 24.5 Примеры — главный инструмент экспертности

**Каждый тезис — конкретный пример.** Без примера тезис = мнение.

**Мини-кейс (формула):** Ситуация → Что сделали → Результат (по предложению).
```
✅ «Интернет-магазин разбил базу на 4 сегмента по RFM.
   Для "спящих" (не покупали 90 дней) запустили реактивацию.
   Из 1200 получателей 180 сделали заказ — конверсия 15%.»
```

**Вместо абстрактного вступления к H2 — сразу конкретная ситуация:**
```
❌ «Интеграция MAX с CRM имеет несколько преимуществ. Во-первых...»
✅ «Клиент пишет в MAX "когда придёт заказ?". Если бот подключён к CRM —
   он отвечает за 2 секунды. Если нет — менеджер видит через 4 часа.»
```

### 24.6 Схемы block-cjm — визуализация пути клиента

**Когда ставить:** статьи о CJM, жизненном цикле клиента, клиентском опыте, онбординге. Любой раздел, где описываются последовательные стадии пути клиента с разными эмоциями.

**Вместо чего:** вместо таблицы «Этап — Точки контакта — Эмоция». `.block-cjm` читается на 70% быстрее за счёт цветового кода.

**HTML-паттерн:**
```html
<!-- wp:html -->
<div class="block-cjm">

  <div class="bcjm-stage bcjm-stage--neutral">
    <div class="bcjm-stage__name">Осознание</div>
    <ul class="bcjm-stage__points">
      <li>Таргетинг</li>
      <li>Рекомендация</li>
    </ul>
    <div class="bcjm-stage__emotion bcjm-em--neutral">😐 Любопытство</div>
  </div>
  <div class="bcjm-arrow">→</div>

  <div class="bcjm-stage bcjm-stage--doubt">
    <div class="bcjm-stage__name">Выбор</div>
    <ul class="bcjm-stage__points">
      <li>Сравнение цен</li>
      <li>Отзывы</li>
    </ul>
    <div class="bcjm-stage__emotion bcjm-em--doubt">🤔 Сомнение</div>
    <div class="bcjm-stage__barrier">Нет сравнения тарифов</div>
  </div>
  <div class="bcjm-arrow">→</div>

</div>
<!-- /wp:html -->
```

**Цветовой код эмоции (верхняя полоса карточки):**

| Класс | Цвет | Когда |
|-------|------|-------|
| `--positive` | зелёный | радость, лояльность, лёгкость |
| `--neutral` | серый | нейтральный интерес, знакомство |
| `--doubt` | янтарный | сомнение, сравнение, раздумье |
| `--risk` | оранжевый | тревога, фрустрация, ожидание |
| `--negative` | красный | боль, разочарование, потеря |

**Правила схемы:**
- Максимум 6 этапов (иначе не читается горизонтально)
- Класс `.bcjm-stage__barrier` — только если есть реальный барьер или проблема на этапе
- Между этапами — `<div class="bcjm-arrow">→</div>`
- Имя этапа в `.bcjm-stage__name` — 1–2 слова
- Точек контакта: 2–3 на этап (не перечислять всё)
- На мобиле схема скроллится горизонтально — норма, не баг
- Стили уже в `wp-custom.css`. В статье: добавить в инлайн `<style>` такой же блок (как CVM и MAX)

## 25. Тултипы (span.explain) — обязательный элемент

### 25.1 Что это и зачем

Тултипы — инлайн-подсказки внутри абзацев. Позволяют ввести термин или источник без сноски. В CVM-статье: **15 тултипов в 5 из 9 секций** — это стандарт качества блога.

### 25.2 HTML-структура

Тултип — **инлайн HTML внутри обычного `<!-- wp:paragraph -->`**:

```html
<!-- wp:paragraph -->
<p>Компания работает с клиентом исходя из его
<span class="explain">пожизненной ценности<span class="explain-card"><strong>CLV / Customer Lifetime Value</strong>Суммарная выручка от одного клиента за всё время отношений. Формула: CLV = средний чек × частота × срок жизни × маржа.</span></span>,
а не среднего по больнице.</p>
<!-- /wp:paragraph -->
```

`explain-card`: `<strong>Полное название</strong>` + текст определения (80–200 символов).

### 25.3 Два типа тултипов

**Тип 1 — Определение термина:**

| Триггер | Карточка |
|---------|----------|
| «CDP» | `<strong>Customer Data Platform (CDP)</strong>Платформа для объединения данных о клиенте из всех источников.` |
| «Bot API» | `<strong>Bot API</strong>Программный интерфейс для создания ботов. Автоматически принимает сообщения, управляет кнопками.` |
| «webhook» | `<strong>Webhook</strong>Способ получения данных от сервера в реальном времени через HTTP-запрос.` |
| «отток» | `<strong>Churn Rate (отток)</strong>Доля клиентов, переставших покупать за период. Снижение на 5 п.п. даёт +25–95% прибыли (Bain).` |

**Тип 2 — Источник цитирования:**

| Триггер | Карточка |
|---------|----------|
| «McKinsey» | `<strong>McKinsey & Company</strong>Один из трёх крупнейших стратегических консультантов мира. Источник: «The growth triple play», McKinsey Quarterly, 2022.` |
| «VK Group» | `<strong>VK Group</strong>Российский технологический холдинг: ВКонтакте, MAX, Mail.ru. Данные — официальные пресс-релизы и VK Company Blog.` |
| «Bain & Company» | `<strong>Bain & Company</strong>Один из трёх крупнейших стратегических консультантов. Авторы концепции NPS.` |

### 25.4 Правила применения

```
✅ При ПЕРВОМ упоминании термина — тултип обязателен
✅ При повторном — только если прошло 2+ секций (H2)
✅ На источниках — при каждом цитировании с данными
✅ В лид-параграфе (.is-lead) — тоже работает
❌ Не давать тултип на общеизвестные слова: «email», «CRM», «сайт»
❌ Не вкладывать тултип в тултип
❌ Не ставить больше 3 тултипов в одном абзаце
```

**Минимум по статье:** ≥5 тултипов на 2000+ слов. Технические темы (MAX, Bot API, интеграции): ≥8.

### 25.5 Лид-параграф: класс is-lead

**Каждая статья начинается с `{"className":"is-lead"}` на первом параграфе:**

```html
<!-- wp:paragraph {"className":"is-lead"} -->
<p class="wp-block-paragraph is-lead">Текст лида...</p>
<!-- /wp:paragraph -->
```

Без `is-lead` лид сливается с остальным текстом. В CVM-статье лид с `is-lead` содержит тултип прямо внутри.

## 26. FAQ-блок: аккордеон + Schema.org FAQPage

FAQ в статьях — **не H3 + параграф**. Используется нативный `<details>/<summary>` с Schema.org. Даёт:
- Аккордеон без JS (HTML5)
- FAQ rich snippets в Яндексе/Google
- Правильную SEO-разметку

### 26.1 WP Block HTML для FAQ-секции

```html
<!-- wp:group {"anchor":"anchor_N","layout":{"type":"constrained"}} -->
<div id="anchor_N" class="wp-block-group">

<!-- wp:heading -->
<h2 class="wp-block-heading">FAQ: ответы на частые вопросы о&nbsp;ТЕМА</h2>
<!-- /wp:heading -->

<!-- wp:html -->
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "Вопрос 1?",
      "acceptedAnswer": { "@type": "Answer", "text": "Ответ 1." }
    },
    {
      "@type": "Question",
      "name": "Вопрос 2?",
      "acceptedAnswer": { "@type": "Answer", "text": "Ответ 2." }
    }
  ]
}
</script>

<div id="SLUG-faq" itemscope itemtype="https://schema.org/FAQPage">

  <details itemscope itemprop="mainEntity" itemtype="https://schema.org/Question">
    <summary itemprop="name">Вопрос 1?</summary>
    <div class="fb" itemscope itemprop="acceptedAnswer" itemtype="https://schema.org/Answer">
      <div itemprop="text">Ответ 1.</div>
    </div>
  </details>

  <details itemscope itemprop="mainEntity" itemtype="https://schema.org/Question">
    <summary itemprop="name">Вопрос 2?</summary>
    <div class="fb" itemscope itemprop="acceptedAnswer" itemtype="https://schema.org/Answer">
      <div itemprop="text">Ответ 2.</div>
    </div>
  </details>

</div>
<!-- /wp:html -->

</div>
<!-- /wp:group -->
```

### 26.2 Правила

```
✅ Минимум 4 вопроса, оптимум 5–7
✅ Вопросы — именно то, что вбивают в поиск (проверить Wordstat)
✅ Ответ начинается с прямого ответа plain language
✅ JSON-LD текст = тот же текст что в itemprop="text"
✅ id="SLUG-faq" — использовать slug статьи
❌ НЕ использовать H3 + <p> для FAQ
❌ НЕ дублировать вопросы из других разделов статьи
```
